Hypercube mode

Last updated: Jauary 10th, 2019

Introduction

MIRO allows the user to load input data, manipulate it, start & stop the GAMS optimization run, inspect the results as well as save, manage and compare different scenarios.

Beside this user interface in its base form, there is another one which allows to configure and evaluate so-called Hypercube jobs. A Hypercube job describes the automated execution of a certain number of scenarios that differ with regard to their parameterization.

Let's take the model "trnsport" as an example again. In the MIRO base mode, among other things, we determine the freight for the transport of goods from the supplier to the market. For the MIP and MINLP model formulation we also can determine the minimum shipment of goods. With a click on solve we start the GAMS run for this parameterized scenario.

But what if we want to solve a multitude of such scenarios with different parameterizations one after the other in order to investigate the behavior of the solutions more closely? To avoid having to configure and solve each scenario individually, there is the Hypercube mode. In this interface a scalar model parameter (a GAMS parameter of dimensionality zero) can be set several times. From this multiple parameterization all possible scenarios are generated, which result from the combination of all set parameters. As a consequence, any number of scenarios can be configured and analyzed at once.

Results of such Hypercube jobs can be stored in the database and inspected directly in MIRO. For this purpose, the Automated Performance Analysis & Visualization tool PAVER was integrated. The direct comparison of scenarios in the scenario view (via split screen / tab view) is also available. Of course, one is free to analyze all the scenarios carried out with the help of external tools as well.

In short, the module presented here is designed to efficiently generate, store and analyze a multitude of scenarios.

Overview

There is little to do to get from the MIRO base mode to the Hypercube interface. We just need to set a simple switch called --miromode=hcube in addition to the already set --miro=launch when starting the WebUI:

Or from the command line: gams trnsport.gms --miro=launch --miromode=hcube

When doing this, all relevant changes in settings regarding appearance and configuration of MIRO are automatically detected and processed.

Visually, the user interface for Hypercube mode hardly differs from the regular one. After starting the application, the following familiar start screen appears.


The navigation menu on the left side now consists of the following sections:

Scenario generation

A scenario is characterized by a particular set of model input (and output) parameters. If, for example, for a run of the trnsport model, the freight is set to 90 and the minimum shipment of goods is set to 100, then this parameterization characterizes the scenario. The Hypercube configuration discussed here makes it possible to configure several such scenarios at once, so that they can then be calculated in a single process.

The configuration of such a Hypercube job for a model is done in the menu 'Configure Hypercube job'. An example is given below.

Configuration of a Hypercube job

If you compare the display of the input data between a single run in the MIRO base mode and the configuration of a Hypercube job, you will notice that the input widgets automatically have changed slightly.

As already mentioned, for parameterizing a single model run in the base mode, we can specify e.g. the freight for the transport and the minimum shipment of goods using a slider. In the Hypercube configuration for these two scalars no simple sliders are available anymore, but sliders with 2 ends, so called range-sliders. In addition, there is a 'step size' for the slider range to be configured (more on stepsize). In addition, the single dropdown menu for selecting the problem type has been replaced by a multi dropdown menu.

Tip:

Input widgets you defined for your scalars in the base mode are automatically expanded in the Hypercube mode (e.g. a slider in base mode is expanded to a slider range in Hypercube mode and a single-dropdown menu is expanded to a multi-dropdown menu etc.).

The convention of automatic scenario generation has the following implications: Unless explicitly specified in the MIRO configuration of the model, each symbol of the input data is included in the Cartesian product from all parameter variations. In other words: All combinations are generated from the configuration selected by the user and each individual combination is saved as a single scenario.

In the example, the slider for 'freight in dollars per case per thousand miles' was set by us to a range from 75 to 150 with a step size of 5. The slider configured in this way leads to 16 different scenarios.

  • Scenario 1: 'freight' = 75
  • Scenario 2: 'freight' = 80
  • Scenario 3: 'freight' = 85
    • [...]
  • Scenario 16: 'freight' = 150

For the scalar 'minimum shipment' the slider was set to a range between 20 and 200 with a step size of 20. The resulting scenarios would therefore be:

  • Scenario 1: 'number of days for training' = 20
  • Scenario 2: 'number of days for training' = 40
  • Scenario 3: 'number of days for training' = 60
    • [...]
  • Scenario 10: 'number of days for training' = 200

The third slider for the scalar parameter beta is not set to a range but to a single value. Due to the non-existent variation of the scalar, only one scenario results from it. The same is true for the last widtget, a multi dropdown menu where we define only the MIP version of our model to be used.

The previously mentioned Cartesian product of all input data variations is now used: It is created from the different variants of the symbols 'freight in dollars per case per thousand miles' (16), 'minimum shipment' (10), 'beta' (1) and 'Select model type' (1) resulting in 16 x 10 x 1 x 1 = 160 individual scenarios.

It gets clear that the number of scenarios to be considered can increase very quickly with the number of scalar parameters and their selected configuration. Depending on how computing intensive the underlying GAMS model is, it may be advisable to configure the scenarios carefully.
Individual scalars can be excluded from inclusion in the Cartesian product. These symbols would then remain unchanged for each scenario created. This can be done in the MIRO configuration for the GAMS model (i.e. in the config.json file) using the option "noHcube": "true" or via Configuration Generator. Example: The scalar 'minimum shipment' in the config.json file is configured like this: 

"mins": {                                      
   "widgetType": "slider",
   "alias": "minimum shipment (MIP- and MINLP-only)",
   "label": "minimum shipment (MIP- and MINLP-only)",
   "min": 0,
   "max": 500,
   "default": 100,
   "step": 1,
   "noHcube": true
}
This confuguration now has the consequence that the scalar can still be selected in the scenario generation, but the double slider is now a single slider again, so it can be assigned only once. Its selection remains identical for each generated scenario.

Multidimensional Symbols:
Unlike scalar values, multidimensional symbols such as sets, parameters and multidimensional singleton sets cannot be varied within a job. They remain identical in every created scenario of a job. For example, if a parameter is part of the input data, the parameter table can be adjusted for a Hypercube job. However, there are no multiple assignments possible for a table, e.g. for a double slider or multidropdown menu. The parameter will therefore be the same for all scenarios of the Hypercube job.

Automated job submission

Once the scenario configuration for a model has been made, it can be completed by clicking on 'Submit jobs'. In the background, now all possible scenario combinations are created. If certain scenarios which we just configured are already stored in the database from a previous run, MIRO asks us whether to process all configured scenarios or only new ones.

Note:

The maximum number of scenarios that can be configured in one step is limited to 10 000.

We now have 2 options:

  1. Automated job submission:
    All scenarios of the Hypercube job are calculated automatically on the machine running MIRO. When the calculations are finished, the results can be loaded into the database.

    2. Job tags: When submitting a Hypercube job automatcally, we can optionally specify job tags. A job tag is an identifier attached to all scenarios of a Hypercube job. With the help of assigned job tags, Hypercube jobs can be uniquely identified, independent of the input and output data of the scenarios. They represent an additional identification feature of a Hypercube job, on the basis of which the database can be searched at a later time. These job tags can also be used to compare different Hypercube jobs with each other with regard to scenario analyses to be carried out.


  2. Manual job submission:
    Manual execution of a Hypercube job. This option is especially interesting if the calculations are not to be performed on the same machine on which MIRO is running, e.g. an extra powerful compute server.
    Job tags can be assigned when uploading the results to the database.

    Note: GAMS version 26.0.1 or higher must be installed on the sysyem running the job.

The following section deals with the automated job submission (option 1). If you choose to do this manually, please follow these steps before you continue reading here.

Import results

When we submit the Hypercube job automatically, it will be executed in the background. Without going into too much detail, what happens here is simple:
A Hypercube file, which contains all scenario configurations, is created by MIRO. This file gets converted into executable GAMS calls, which are then processed. The script which is responsible for this conversion is located in the /resources folder of the MIRO installation directory and can be customized as needed. More about these steps in the section on manual execution of a Hypercube job.

Once a Hypercube job has been submitted, it is shown in the Import results section which can be accessed via the sidebar menu. There, all jobs are listed that are still running or whose results have not yet been imported.

The overview shows for each started job the owner, the submission date and the (optional) specified job tag(s). In addition, the current status ("running", "completed" or "scheduled") is visible. The current log of a run can be accessed at any time (see image below) and results that have not yet been loaded into the database or are still pending can be discarded. As soon as a job has finished, the results can be loaded into the database with a click on import.

Manual job submission

If we want to execute a Hypercube job manually, MIRO generates a so-called Hypercube file (<modelname>.hcube). This file contains all scenario configurations as individual (not yet executable) GAMS calls. A Zip archive including this file is downloaded by clicking on Submit job. The ZIP archive has the following contents:

  • static folder
    This folder contains all non-scalar (thus multidimensional) input data of the corresponding GAMS model. As explained here, this data remains the same for all scenarios of a Hypercube job.
  • trnsport folder
    This folder is a copy of the directory of the underlying GAMS model. If files are included in the model using relative paths, they should be located in this folder.
  • hcube_submission.gms
    Skript which is used in the automated job submission to convert the Hypercube file into executable GAMS calls. The folder structure of the downloaded ZIP file corresponds to the structure assumed in the hcube_submission.gms file.
  • trnsport.hcube
    Hypercube file. Contains the GAMS calls for the scenarios and all scalar information set for the Hypercube job in MIRO.

The crucial point in the procedure for the manual execution is to create a script which can be executed by GAMS from the information given in the .hcube file. The mentioned file hcube_submission.gms can be used to create this script. However, if adjustments are to be made, these must now be taken into account.

Let's first have a brief look at the trnsport.hcube file.

Each row of the file contains the information needed for exactly one scenario.
Example: In the first line of the shown Hypercube file it says:

2d30a49f5fb10b2aa8aab9e133db4f079c8c753a23ce6f67f1086b572f57ba7c: gams trnsport.gms --HCUBE_STATIC_a=8bc030208cf6482932608872c5681310 --HCUBE_STATIC_b=a5126cbcb9f17e29d4b0e49aaf1087ff --HCUBE_STATIC_d=50ae5383d3e43c0c902144abbf0b3e8e --HCUBE_SCALAR_f=75 --HCUBE_SCALAR_mins=20 --HCUBE_SCALAR_beta=0.97 --type="mip" --MIRO=RUN --HCUBE=1 execMode=0 lo=3 trace="_sys_trace_trnsport.trc" traceopt=3 idir1="..\..\trnsport"

The following information is included:

  • Hash value:
    A row always begins with an individual, so-called hash value which is used to uniquely identify a scenario. Those values are calculated / hashed using the sha256 algorithm based on the respective line of the Hypercube file. Results of a scenario must end up in a directory named after the hash value.
  • GAMS call:
    After the hash, the basic GAMS call (in this example: gams trnsport.gms) follows.
  • Scenario input data:
    A scenario is calculated using scenario-specific input data. This input data consists of the parameterization of scalars, parameters, command-line parameters, and so on, which have been configured in MIRO. All information about model input data coming from MIRO is contained in the Hypercube file in some way. Since a model run for a scenario is started via a GAMS call, this call must also contain all information about the corresponding input data.
    • Static data:
      Non-scalar / multi-dimensional input data such as parameters are used as fixed files for all scenarios of a Hypercube job. In the Hypercube file such static files are specified with --HCUBE_STATIC_<parameter-name>=<hash>.
      Example: In the trnsport model, there is a parameter a which defines the capacity of plants. When this parameter is used for a Hypercube job, in the corresponding file it says --HCUBE_STATIC_a=131a04011c1c227dccbbbc7e742073e2.
      In the automatic execution of Hypercube jobs this information is used internally. In manual execution, the string can be ignored. However, it must be ensured that the static files are correctly named and located in the right directory static.
    • Command line parameters:
      Double-dash parameters and GAMS options which are set via MIRO are already in the correct format for a GAMS call.
      Examples:
      --type=lp → A compile-time variable "type" set to lp
      lp=CPLEX → GAMS option lp to select a LP solver set to CPLEX
      (the latter is not part of the above example).
    • Scalars:
      Scalars can usually not be defined via a GAMS call, but are set in the model itself. In order to be able to set scalars via a Hypercube job, the following happens:
      If a scalar is set, this is recorded in the Hypercube file as --HCUBE_SCALAR_<scalar-name>=<scalar-value>, e.g. --HCUBE_SCALAR_f=90 for scalar f and a value of 90. To continue this example, if a GAMS model is called with --miro=run and the compile-time variable --HCUBE_SCALAR_f=90, a scalar f is searched in the symbol declarations of the underlying GAMS model and, if declared, set to the value of the compile-time variable, i.e. 90.
  • Trace file command:
    The trace file should be created for each scenario. This provides the integrated analysis tool PAVER with important information for performance analysis. Unless otherwise set, the GAMS command for creating this trace file is included for each scenario in the Hypercube file (here: trace=_sys_trace_trnsport.trc traceopt=3)
  • When calculating Hyperube jobs, a GAMS call must be initiated with the compile-time variable --miro=run. This leads to the inclusion of a MIRO specific file, which is especially essential for the input data handling of a scenario. This includes the next double-dash parameter --HCUBE=1. Like --miro=run it should not be changed.
  • The GAMS options execMode=0 and lo=3 are relevant for the automated execution of a Hypercube job and can be changed or ignored for the manual execution.

Now we need to prepare the information given in such a way that GAMS can be executed (thus a scenario can be calculated) with it. An important part of this is the handling of input data.

String converison:
Since the strings in the form in which they are available in the Hypercube file are not calls that can be interpreted by GAMS due to the hash value at the beginning, the information about the individual calls must be extracted. Probably the easiest way to do this is to take each scenario-specific GAMS call and execute it. This can be done with the help of the hcube_submission.gms script or an own method.

Note:

Irrespective of whether the provided script or a custom method is used, the rules for a trouble-free import of results must be observed.

If the scenarios are to be executed on a external compute server, it is now time to take all necessary precautions, which have to be considered in the script (e.g. folder structure on computer, login data, etc.). In short, the jobsubmission script must be adapted to the environment in which it will finally be executed.

Compute server:
If a script that can be interpreted by GAMS has been created from the Hypercube file (e.g. as described above), it can be executed. If the model calculations are to be performed on a compute server, this can now be done.

Import results:
At the end of the calculations, the results should be available in the form of another ZIP file. There are rules for the upload of such a zip file, which have to be followed to be accepted by MIRO as valid scenario data.

  • Results have to be separated according to scenarios:
    Each scenario has a separate folder in the root directory of the ZIP file. This folder must be named after the hash value of the scenario. It contains the calculated result / output data.

For each scenario there are some files that must exist to be valid, and some that are optional. The mandatory files for every single scenario are:

  • scalars.csv
    This file is generated automatically when running a model with --miro=run. The --miro=run is always set for a scenario in the downloaded <modelname>.hcube Hypercube file. The scalars.csv file contains all scalar values which are declared between the $onExternalInput / $offExternalInput tags (or 'UIInput: ' in the explanatory text of a scalar) in the underlying GAMS model.
    The header line of the scalars.csv file always must be "Scalar,Description,Value" where Scalar stands for the GAMS Symbol name, Description for the optional explanatory text and Value contains the value set by the user from within MIRO.
  • scalars_out.csv
    This file contains all scalar output values specified between the $onExternalOutput / $offExternalOutput tags (or 'UIOutput: ' in the explanatory text of a scalar). Like the scalars.csv file it is generated automatically after a successful model run. So this is the counterpart to the scalar input values in scalars.csv.
    Note: If, for whatever reason, a model calculation is aborted for a scenario, then this file scalars_out.csv is not created. However, since the file is mandatory, all scenarios whose calculations are aborted (intentionally or unintentionally) are ignored during a database upload.
  • _sys_trace_<modelname>.trc
    Trace file of the scenario run. This file contains numerous run statistics which are used for the the integrated performance analysis tool PAVER. The downloadable Hypercube script already contains the GAMS command for the creation of the trace file for each scenario. More information on the trace file can also be found here.

These three files are usually generated automatically when solving a scenario with the compile-time parameter --miro=run. Their format is internally defined by MIRO and should not be changed. If scenario data does not contain all mandatory files, the data for this scenario is not stored in the database.

In addition to the mandatory files mentioned above, there may be other files in the result folder of a scenario. If --miro=run is set, a CSV file with results is created for each output parameter declared in the GAMS model between the $onExternalOutput / $offExternalOutput tags (or 'UIOutput: ' in the explanatory text). For example, when the Parameters results and performance are declared between those tags, the CSV files results.csv and performance.csv are generated after the model calculations. For these files the following rules apply:

  • Result files of output parameters are not mandatory, but optional for the upload into the database.
  • The results must be available in CSV format.
  • CSV files must be named after the parameters declared in the GAMS model between the $onExternalOutput / $offExternalOutput tags (or 'UIOutput: ' in the explanatory text). Each output parameter declared there must be stored in a separate CSV file. Exceptions are scalar values, see here.
  • It is mandatory that the header line of a CSV file is identical to the corresponding parameter declared in GAMS.
    Example: In the trnsport model, the output parameter schedule(i,j,scheduleHdr) is declared between the $onExternalOutput / $offExternalOutput tags. The resulting schedule.csv contains the header row "canning plants,markets,cap,demand,quantities", which corresponds to the explanatory texts of the indexes i and j as well as the set elements of the declared header scheduleHdr (see image below). The conventions concerning the selection of the headers can be found here.

    [...]
                                        
$onExternalOutput
Variable z 'total transportation costs in thousands of dollars';
$offExternalOutput
Positive Variable x;

[...]

As already mentioned, CSV files are usually generated automatically. However, optional files can be deleted manually so that they are not included in the database.

Scenario upload:
Once the desired scenarios have been calculated, the next step is to load the results into the database. For this purpose the module import results is used. In the overview, which shows the automated generation and loading of results, click on Manual import at the bottom right.

Upload Zip file with results:

Again, for a trouble-free upload, the scenario data must be available in a zip file. Each calculated scenario is located in a separate folder, which is named after the hash value of the scenario. In addition to the input and output data of the scenario (in the form of CSV files), the trace file of the GAMS run is also located there.

When uploading the zip file you can optionally specify job tags.

With a click on 'upload', the files to be uploaded are evaluated with regard to their structure compatibility and, if there is nothing to complain about, stored in the database.

Analyses

Scenario selection

Scenarios that have been imported into the database can be loaded at a later time for subsequent analyses. For this purpose the module "load scenarios" is used, which represents a graphical representation of a database query. If not all available scenarios are to be displayed, the query can be strongly individualized. All scalar input values that can be set by the user to uniquely identify a scenario can be selected as filters.

Example:
In the following example from the demo model trnsport 3 scalar input values are integrated into the WebUI:

$onExternalInput 
Scalar f    'freight in dollars per case per thousand miles' / 90 /
       minS 'minimum shipment (MIP- and MINLP-only)' / 100 /
       beta 'beta (MINLP-only)' / 0.95 /;
$offExternalInput

In the database query, you can search for exactly these scalar data of a scenario.
Next to those scenario specific filters, the optionally defined job tags of a Hypercube job and the date of creation can be used in a database query as well. Database queries can be combined with the help of the Boolean Operators AND and/or OR.

If all desired filters are set, the database query is started with a click on 'query database'. The results are then displayed in the lower screen area. There the user can select scenarios for the analysis.
If this is done the user is asked to chose an analysis method:

  • Interactive mode:
    Internal scenario comparison. Compare different scenarios directly in split view or tab view. Best suited for a small number of scenarios to compare (maximum number of scenarios allowed: 10).
  • Paver:
    Automated Performance Analysis. Suitable for a large number of scenarios to be compared with regard to defined performance criteria.
  • Download data:
    Download of scenario data to be analyzed with an external analyzing method.

The various analysis methods are described in detail below.

Internal Analyses

Scenario comparison:

The scenario comparison module is used to directly compare the input and output data of a small number of scenarios. The data which is displayed here corresponds to those GAMS Symbols which were marked within the corresponding GAMS model with the $onExternalInput / $offExternalInput and $nExternalOutput / $offExternalOutput tags, i.e. those symbols which were marked for inclusion into the WebUI. Conversely, this means that all parameters and scalar values which shall be compared in the WebUI must be marked in the GAMS model accordingly. More information about these tags can be found here.
The internal scenario comparison module has already been explained here.

PAVER:

PAVER is an Open Source Environment for Automated Performance Analysis of Benchmarking Data. This tools is meant to simplify the task of solver performance data comparison and visualization by providing a variety of performance measurement tools with convenient and easy ways to represent (e.g. objective values, solution times, and result codes from multiple runs in order to summarize, gain insight, or detect inconsistencies) [Bussieck, Michael R., Steven P. Dirkse, and Stefan Vigerske. "PAVER 2.0: an open source environment for automated performance analysis of benchmarking data." Journal of Global Optimization 59.2-3 (2014): 259-275, link].


PAVER independently performs a variety of analyses regarding the performance of selected scenarios. A scenario is defined by the individual parameterization of its input data. In addition to the input data, the results, i.e. the output data, are of course particularly interesting and relevant for a performance analysis. Paver uses this output data for its evaluations. So far, PAVER can read comma separated value (csv) files that contain in each line the outcome of a solver run (instance name, solver name, options file name, instance statistics, solving statistics, ...). With GAMS, files in this format can be generated with the options trace=file.trc and traceopt=3 which is done in the Hypercube file by default.


Before the start of the PAVER analysis, the user is asked to define the criteria on the basis of which the scenarios are to be compared with each other. Basically, scenarios can be compared on the basis of:
  • Scalar input data for the corresponding GAMS model
  • Scalar output data for the corresponding GAMS model
  • All characteristics contained in the trace file
  • Job tags optionally assigned by the user for each Hypercube job
  • Date of creation of a Hypercube job
  • Owner of a Hypercube job

Example:
For the already discussed model pickstock the user can define the scalar compile-time parameters for the maximum number of stocks to select (alias 'MaxStock') and the number of training days (alias 'TrainingDays') as input data. These parameters are therefore comparable input data in PAVER.
Selectable compile-time parameters and GAMS options for a model run via the WebUI:

  • MaxStock
  • TrainingDay

In addition to such input data of the model, scalar output values are also calculated in the GAMS model:

  • last date of training period
  • Stock symbol header
  • fund header

All these compile-time parameters, GAMS options and scalar values are used as such in the corresponding GAMS model. More information about using scalars, singleton sets, compile-time parameters and GAMS options via WebUI can be found here.

External Analyses

If external analyses are to be carried out instead of the WebUI's integrated analysis methods, this is also possible. For this purpose, the WebUI offers the possibility to download a selection of scenarios so that the corresponding data can be used elsewhere. The structure of the downloaded ZIP file is the same as in those files which can be imported into the database. Each selected scenario is stored in a separate, hashed folder. Such a scenario folder contains both input and output data in CSV format. The trace file of the corresponding GAMS run now also is in CSV format.
