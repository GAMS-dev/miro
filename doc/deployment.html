App deployment

Introduction

This chapter deals with the deployment of a MIRO application. All configuration should be completed beforehand.

Development vs. Deployment

During the development process of a MIRO application, the GAMS symbols and graphics to be displayed and thus the corresponding configuration files are constantly changed. These adjustments are no longer possible once an application has been deployed.

Note:

The configuration of a deployed MIRO application can not be changed anymore! Only when the app was started in development mode again and the .miroconf files were updated, changes become effective.

When starting a deployed MIRO app, instead of the JSON files the binary files <modelname>_<version_number>.miroconf and <modelname>_<version_number>_hcube.miroconf are used for the startup. These configuration files are only created/updated when an app is started in development mode, i.e. via GAMS Studio or the command line. Thus, they are always up to date with the last launch in that particular mode (Base mode or Hypercube mode). If the MIRO app has been changed, e.g. the configuration of a new graphic or the specification of a new GAMS symbol, these changes are not taken into account when starting your app in deployment mode, since these changes are only performed in the JSON files. In order to update the deployed app, the .miroconf files must be recreated. For this you first to re-launch your app in development mode. Note that you need to launch the Hypercube mode explicitly for the <modelname>_hcube.miroconf file to be refreshed!

Note:

There is one binary configuration file each for Base mode and Hypercube mode. Both are only updated when the corresponding mode is started in development mode.
