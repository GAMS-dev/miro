Model deployment

Last updated: Jauary 14th, 2019

Introduction

After testing the demo applications, let's now prepare an existing GAMS model for use with the WebUI. An essential feature of GAMS WebUI is that this can be done in a fast, easy and automated way. For a better understanding it may be helpful to have a look at the GAMS code of the demo applications.

The well-established trnsport model is included as fully integrated demo application in the WebUI. However, for illustration purposes we now take the original trnsport.gms model from the GAMS model library. This model will be adapted for use with the WebUI in a few steps and within a few minutes.

If you want to go through the step by step instructions with the trnsport model, then download the corresponding file from the Model library and put it in a directory of your choice.

Turn your GAMS model into a WebUI

To be able to use the WebUI, an interface must be created between WebUI and the GAMS model to be deployed. This interface primarily consists of the information which symbols of the GAMS model should be visible (and changeable) by the user from within the WebUI. Those GAMS symbols must be flagged in the GAMS code.

Note:

A GAMS model still can be used in the normal way (via GAMS Studio, GAMSIDE or the console) after the following adjustments

In principle, there are only three steps to be done:

  1. Flagging input data with / or UIInput:
  2. Flagging input data with / or UIOutput:
  3. At the very end of the model include ($libinclude) the file webui with
We will examine these steps more closely in the following.

Model adjustments

Every GAMS model can be launched inside the WebUI in just a few steps. First, those GAMS symbols that you want to be visible in the interface need to be declared. This can be done by framing the corresponding symbols with the tags / for input data and / for output data. These tags can be used multiple times within a model. Alternatively one can write UIInput: or UIOutput: at the beginning of the explanatory text of a GAMS symbol to include this symbol in the WebUI. Symbols which are not tagged with one of these markers wonâ€™t be visible in the Web Interface, but they are used within the model run in the background.

Note: Those symbols which shall be displayed in the WebUI must have an explanatory text in case you are using a GAMS version below 25.2.

The code below shows a simple example for this:

$onExternalInput
Parameter
   symbol_1(dim1, dim2)  'Displayed in WebUI (model input data)'
   symbol_2(dim3)        'Displayed in WebUI (model input data)';
$offExternalInput

Parameter
   symbol_3(dim4)        'NOT displayed in WebUI'
   symbol_4(dim5)        'UIOutput: Displayed in WebUI (model output data)';

Here, the parameters symbol_1 and symbol_2 are declared between the mentioned tags $onExternalInput and $offExternalInput. They will be visible in the WebUI (e.g. as tables).
symbol_3, on the other hand, was declared outside of said keywords. This parameter will not be visible in the WebUI, however, it is still used as normal parameter in the GAMS model.
symbol_4 will be displayed in the WebUI as output data because its explanatory text starts with UIOutput.

The following Symbols can be communicated via the WebUI:

  • Parameters / Tables
  • Scalars
  • Singleton Sets
  • GAMS Options
  • Compile-time Variables

The configuration of GAMS Options and Compile-time Variables differs from the other symbols. Learn more about this here.

Let's try this for our demo model trnsport now. This is what the input section of the model looks like when we have loaded it from the Model library:

Set
   i 'canning plants' / seattle,  san-diego /
   j 'markets'        / new-york, chicago, topeka /;

          san-diego  600 /

Meaningful input data for the WebUI could be the symbols a, b, d and f. In order to make these symbols visible in the WebUI, we now mark them with said tags:

Set
   i 'canning plants' / seattle,  san-diego /
   j 'markets'        / new-york, chicago, topeka /;
   
$onExternalInput
Parameter
   a(i) 'capacity of plant i in cases'
        / seattle    350
          san-diego  600 /

   b(j) 'demand at market j in cases'
        / new-york   325
          chicago    300
          topeka     275 /;

Table d(i,j) 'distance in thousands of miles'
              new-york  chicago  topeka
   seattle         2.5      1.7     1.8
   san-diego       2.5      1.8     1.4;

Scalar f 'freight in dollars per case per thousand miles' / 90 /;
$offExternalInput

As output data the final objective function value can be used, for example. Since variables cannot be displayed in the WebUI, we define a new scalar for this purpose, mark it as output value for the WebUI and assign it the final objective function value after the completed calculations:

[...]
                                        
Model transport / all /;

solve transport using lp minimizing z;
display x.l, x.m;


$onExternalOutput
Scalar total_cost 'total transportation costs in thousands of dollars';
$offExternalOutput

total_cost = z.l;

With this we have defined both input and output data for use in the WebUI.
A last step now is to include a file webui.gms by a at the very end of the GAMS model. With the help of this file all model specific configuration files are created automatically.

[...]
                                        
$onExternalOutput
Scalar total_cost 'total transportation costs in thousands of dollars';
$offExternalOutput

total_cost = z.l;

$libInclude webui

Note: The file webui.gms is copied into the GAMS library include directory (by default, this is the directory inclib in the GAMS system directory) during the WebUI installation and can be included with a $libinclude.

That's it. The model is now ready in a first version for use with the WebUI.

Note:

You can also use the WebUI to set GAMS options and compile-time variables for a model run. More about this is explained here.

Generating model-specific configuration files

In the next step the model-specific configuration files for the Web Interface are created.

  • Run the model with the compile time parameter --webui=compile.

    Warning:

    If you use a GAMS $call inside a model you need to pay attention to the path of external files you might include in this call. This is due to the different working directories depending on whether a GAMS model is run from the WebUI or not. The GAMS WebUI is providing you the path to your model file via the GAMS option idir1. This means that in order to e.g. import an Excel file test.xlsx, the GDXXRW tool needs to be called like this: $call gdxxrw i="%gams.idir1%test.xlsx". Alternatively, you may use absolute paths.

  • After the run, a new folder named conf is automatically created in your model directory. This folder contains the file GMSIO_config.json. In this file you can find the configuration of the input and output symbols in JSON Synax, which were previously defined in the GAMS model between the $on/offExternalInput and -output tags.
    The model folder also contains an Excel file with the same name as your model file. This Excel file contains all the relevant input data that was extracted from your model and can be used to populate your WebUI.

    Note:

    In case you don't find an Excel file with the data, please make sure the xlsxwriter Python module was successfully installed (c.f. here).

  • The very basic configuration is now done. You can start the WebUI by running the model again, this time with --webui=launch.
    The first time this is done, all R packages needed for the WebUI will be automatically installed. This may take a while. When the installation is complete, the WebUI opens with the default browser of the operating system.


Congratulations! A first version of your new WebUI is now up and running!

With the basic configuration you can upload Excel files, modify them from within the interface and start or stop a GAMS job. You also have access to the log and lst files of GAMS. When the model run has finished successfully, the results can be examined. In the default configuration, tables and pivot tables are generated. Since the WebUI is connected to a database, you can store results there, reload data and compare scenarios.

Visualization conventions

There are a few conventions for the representation of symbols in the WebUI, which are explained in this section.

Visualization of parameters - tables

For the examples below we use the demo model Pickstock. In this model the following input data has been marked for use within the WebUI:

$onExternalInput
Set date   'date'
    symbol 'stockSymbol';
    
Parameter price(date,symbol) 'Price';

Scalar maxstock        'maximum number of stocks to select'  / 2  /
       trainingdays    'number of days for training'  / 99  /;
$offExternalInput

Here, the parameter price as well as the scalars maxstock and trainingdays are declared between the mentioned tags $onExternalInput and $offExternalInput.

When starting the WebUI via --webui=launch and loading the newly created excel spreadsheet into the WebUI, the following input screen appears:

In the default WebUI configuration, each index of a GAMS Parameter is shown in a separate column. The last column contains the parameter values. With this configuration, each set element of an index will be visible in a separate row. In the given example the Parameter price has two indices date and symbol. Each index is shown in a separate column named after the explanatory text of the corresponding set (or in case no explanatory text is available, the name of the symbol).

Sometimes it can be useful to show each set element of an index in a separate column of the table. For the parameter of the example above this would mean to have a column not for the set symbol ("stockSymbol") itself but for all its elements AAPL, AXP, BA, etc.
In order to achieve this type of pivot functionality, a minor change must be made to the declaration of the corresponding symbol, namely a keyword must be included.

Keywords are: header, hdr and pivot

One can either:

  • Change the name of the set which should be displayed as header so that it ends with a header-keyword. To avoid renaming the set for the entire GAMS model, you can declare an alias for the set afterwards.
    In the example this is done with symbol which is changed to symbol_header. 
    $onExternalInput
Set date   'date'
    symbol_header 'stockSymbol';
    
Parameter price(date,symbol_header) 'Price';

Scalar maxstock        'maximum number of stocks to select'  / 2  /
       trainingdays    'number of days for training'  / 99  /;
$offExternalInput
Alias(symbol_header,symbol);

  • Change the explanatory text of the set which should be displayed as header so that it ends with a header-keyword. 
    $onExternalInput
Set date   'date'
    symbol 'stockSymbol hdr';
    
Parameter price(date,symbol) 'Price';

Scalar maxstock        'maximum number of stocks to select'  / 2  /
       trainingdays    'number of days for training'  / 99  /;
$offExternalInput

  • Change the explanatory text of the parameter which should have a header so that it ends with one of the header-keywords. When doing this, the elements of the last parameter index will be displayed in the header. (In the example this is done with price). 
    $onExternalInput
Set date   'date'
    symbol 'stockSymbol';
    
Parameter price(date,symbol) 'Price pivot';

Scalar maxstock        'maximum number of stocks to select'  / 2  /
       trainingdays    'number of days for training'  / 99  /;
$offExternalInput

The result of all these alternatives is shown in the example below.

Each element of symbol (AAPL, AXP, BA, etc.) is now represented in its own column.
Note: By renaming the the set symbol to e.g. symbol_header or change its explanatory text, we achieve the same thing. In both variants it would be used as header in every parameter in which it is defined. If you only want to specify the header of one single parameter, independent of others, you need to use the header keyword at the end of the explanatory text of exactly this parameter (in this case the explanatory text of parameter price).

A table width quickly increases when the number of set elements of the header index gets higher. A number of, for example, 30 Stocks to be represented in a table would lead to 30 stock columns. In this case the default configuration for a table as shown first might be better.

If a Parameters with multiple value columns has to be displayed within the WebUI, a manual definition of the header is mandatory. This is done in the demo model kport for the Parameter PData. By defining the Index phdr as last index of PData, the set elements Demand and Production Time are shown as value columns in the WebUI table.

$onExternalInput
Set 
   phdr   'Product data headers' / Demand, 'Production Time' /
   s      'Scenario'             / s1,s2  /
   pP     'Product'              / L1*L37 /;
    
Parameter PData(pP,s,phdr) 'Product data';
$offExternalInput

Visualization of scalar values

In addition to one- or multidimensional GAMS parameters, 0-dimensional GAMS symbols such as scalars and singleton sets can also be communicated via the WebUI. The handling of these symbols differs from the previously discussed parameters.

Note:

Scalars and singleton sets are collected in a separate table in the interface by default.

They are collected in a separate table in the interface by default. This means that all scalars defined in the GAMS model within the $on/offExternalInput and -output are shown in the Web interface in one single 'scalars' table. The same is also true for scalar output parameters. These are summarized in the results section in a separate 'scalars_out' table.

Here we have declared the scalar input data 'maxstock' and 'trainingdays' (see image above) between the input tags. In the WebUI, they are displayed aggregated in a table.

Not only parameters, but also scalar values can be displayed with appealing and useful graphics (we call them widgets in this case). Scalar values are great for representation as a slider, dropdown menu, checkbox or date selector. You can learn more about this here.

Command line Parameters

In addition to the "standard" GAMS symbols like parameters and scalars, command line parameters can also be set as input data via the WebUI, more precisely this is true for GAMS Options and Compile-time Variables. The configuration of these Command line Parameters differs from that of the other symbols. Parameters, Scalars and Singleton Sets are declared directly in the GAMS code and can therefore be tagged with / . GAMS options and compile-time variables are specified in GAMS via the command line and can therefore not be tagged in the GAMS code. To be able to use them in the WebUI, they must be specified in the config.json file. Read more here.

Custom configuration

If required, the web interface of the GAMS model can now be further customized. The GAMS WebUI comes with various pre-implemented graphing options: at the moment these are pie charts, bar charts, line charts, scatter plots and histograms. However, an R API is provided so even highly customized graphics can be generated.
