Installation

GAMS Installation

The GAMS WebUI is a graphical user interface (GUI) for your GAMS models that runs inside your browser. This interface allows you to to feed your models with input data as well as generate and view the results of your optimization. Various graphing tools support your data visually to get the most insight out of your GAMS models.

To generate a GAMS WebUI for your model the following software is required: To be able to use the WebUI, a current GAMS version (25.1.2 or later) is required. You can download it from https://www.gams.com/download/.

Note:

Some of the example models included in the GAMS WebUI installation require additional Python packages. These packages should get installed automatically once you run the respective model for the first time. However, in case they do not install properly, try installing them manually: from the command line navigate to your GAMS installation folder and then to the subfolder: GMSPython\Scripts (e.g.: cd C:\GAMS\win64\25.1\GMSPython\Scripts). To install the respective package, enter: pip install <package_name>. Below you find a table with the python packages required for the different example models.

GAMS WebUI Installation

To install the GAMS WebUI and its components, run the GAMS WebUI setup. Since a current R installation is required, the installer for Windows checks for an existing R version (3.5.1 or later) and installs it, if necessary. On Mac and Linux, you currently need to install R manually (download here) in case it is not yet installed on your system.

First steps

To get a feeling for the use of the WebUI, it is recommended to have a look at the demo applications first. After a successful installation, these demo applications are located in C:/Users/<User_Name>/Documents/GMSWebUI_Examples/. To start a demo application, you need to run the corresponding GAMS model (e.g. GMSWebUI_Examples/trnsport/trnsport.gms) with the compile-time parameter --webui=launch. This can be done from within the new GAMS Studio, the GAMSIDE or using the command prompt (Note: To use the command prompt, GAMS must be in the PATH environment variable of your operating system):

cd path/to/gms_file && gams trnsport.gms --webui=launch

Note:

If you run a model with --webui=launch all required R packages will be loaded. If you use the WebUI for the first time, the installation of these packages may take a while.

Once all the packages have been installed, the GAMS WebUI will launch. Below, you see the input screen of the WebUI as it presents itself after you launched it for the trnsport model.

The screen is essentially divided in two:

Navigation bar:
On the left side you can find the navigation bar. Here you can navigate between the categories "Input", "Output", "GAMS interaction" and "Advanced Options". In addition, the buttons for loading input data and for starting a GAMS run are located here.

Main window:

The main window is the heart of the WebUI. Here you can see your data in the form of tables and graphs.

  1. Import data
    To import model input data, click on the "Load data" button and navigate to the folder of the model currently used (e.g. Documents/GMSWebUI_Examples/trnsport/). You should find a trnsport.xlsx file there. After the import, you can adjust the input data.
  2. Model run
    A click on the "Solve model" button starts the model run. During the calculations, the WebUI switches to the section "GAMS interaction". There you can see the current GAMS log and lst files. After the run, the view changes to the "Output" section.
    Note: Since the models of the demo applications are solved very quickly, this step is sometimes hardly visible. The menu entry "GAMS interaction" can also be viewed at any time after a model run.
  3. Results
    In the "Output" section the result parameters are visualized in tabular form. With a click on the button in the upper right corner you can switch between tabular and graphical data visualization. All scalar results are summarized in the tab "Scalars". The button enables the user to download the model input and output data.
  4. Advanced Options
    Besides the model input and output data all temporarily created files of the model run like the lst and log files can be downloaded separately or as ZIP file in the "Advanced Options" menu.

Turn your GAMS model into a WebUI

After testing the demo applications, it's time to get started yourself. For a better understanding it is recommended to have a look at the GAMS code of the sample applications for the next steps.

Note:

A GAMS model still can be used in the normal way (via GAMS Studio, GAMSIDE or the console) after the following adjustments

Model adjustments

Every GAMS model can be launched inside the WebUI in just a few steps. First, those GAMS symbols that you want to be visible in the interface need to be declared. This can be done by framing the corresponding symbols with the tags / for input data and / for output data. These tags can be used multiple times within a model. Alternatively one can write UIInput: or UIOutput: at the beginning of the explanatory text of a GAMS symbol to include this symbol in the WebUI. Symbols which are not tagged with one of these markers won’t be visible in the Web Interface, but they are used within the model run in the background.

Note: Those symbols which shall be displayed in the WebUI must have an explanatory text in case you are using a GAMS version below 25.2.

Visualization of parameters as tables

In the default configuration of a GAMS parameter, each index of a Parameter is shown in a separate column. The last column contains the parameter values. With this configuration, each set element of an index will be visible in a separate row. In the example below the Parameter price has two indices date and symbol. Each index is shown in a separate column named after the explanatory text of the corresponding set (or in case no explanatory text is available, the name of the symbol).

Sometimes it can be useful to show each set element of an index in a separate column of the table. For the parameter of the example above this would mean to have a column not for the set symbol itself but for all its elements AAPL, AXP, BA, etc.. To achieve this, a minor change in the declaration of the corresponding set has to be made.

One can either:

  • Change the name of the set so that it ends with the header-keyword: hdr (to avoid renaming the set for the entire GAMS model, you can declare an alias for the set afterwards).
  • Change the explanatory text of the set so that it ends with the keyword hdr. This is shown in the image below.

Note that a table width in this way quickly increases when the number of set elements of the last index gets higher. A number of for example 30 Stocks to be represented in a table would lead to 30 stock columns. In this case the default configuration for a table as shown first might be better.

If a Parameters with multiple value columns has to be displayed within the WebUI, a manual definition of the header is required. In the picture below this is done for the Parameter PData. By defining the Index phdr, the set elements Demand and Production Time are shown as value columns in the WebUI table.

Note:

Scalars and singleton sets are by default collected in a separate table in the interface.

If all input and output symbols for the WebUI have been flagged, in a last step a file webui.gms has to be included by a at the very end of the GAMS model. With the help of this file all model specific configuration files are created automatically.

Generating model-specific configuration files

In the next step the model-specific configuration files for the Web Interface are created.

  • Run the model with the compile time parameter --webui=compile.

    Warning:

    If you use a GAMS $call inside a model you need to pay attention to the path of external files you might include in this call. This is due to the different working directories depending on whether a GAMS model is run from the WebUI or not. The GAMS WebUI is providing you the path to your model file via the GAMS option idir1. This means that in order to e.g. import an Excel file test.xlsx, the GDXXRW tool needs to be called like this: $call gdxxrw i=%gams.idir1%test.xlsx. Alternatively, you may use absolute paths.

  • After the run, a new directory named conf was automatically created. This directory contains the file GMSIO_config.json. In this file you can find the configuration of the input and output symbols in JSON Synax, which were previously defined in the GAMS model with the $on/offExternalInput and -output tags. The model folder also contains an Excel file with the same name as your model file. This Excel file contains all the relevant input data that was extracted from your model and can be used to populate your WebUI.

    Note:

    In case you don't find an Excel file with the data, please make sure the xlsxwriter Python module was successfully installed (c.f. here).

  • The very basic configuration is now done. You can start the WebUI by running the model again, this time with --webui=launch.
    The first time this is done, all R packages needed for the WebUI will be automatically installed. This may take a while. When the installation is complete, the WebUI opens with the default browser of the operating system. With the basic configuration you can upload Excel files, modify them from within the interface and start or stop a GAMS job. You also have access to the log and lst files of GAMS. When the model run has finished successfully, the results can be examined. In the default configuration, tables and pivot tables are generated.

Custom configuration

If required, the web interface of the GAMS model can now be further customized. All the configuration files for your WebUI are JSON files located in the directory: <modeldirectory>/conf. The GAMS WebUI comes with various pre-implemented graphing options: at the moment these are pie charts, bar charts, line charts, scatter plots and histograms. However, an R API is provided so even highly customized graphics can be generated. All this configuration is handled via the file: config.json. To generate this file, we recommend using our Configuration Generator. However, you may also use the JSON schema file which is located in <GAMSroot>/GMSWebUI/conf/. All the configuration files are validated against a schema, so you may use these schemata to learn how your JSON files should be structured.

Note:

You can generate this JSON file using our Configuration Generator

Advanced configuration

If you want to visualize your data in a way that is not yet pre-implemented, there is the possibility to implement it yourself via an R API the WebUI comes with. The R code for these custom graphics must be stored in the directory: <modeldirecory>/customRenderer/. To learn more about this, read the documentation on custom renderers.

Additional Features

Manage multiple scenarios

Did you know that you can connect the GAMS WebUI to a database? This way, different input dataset and results, called scenarios, can be loaded, modified, stored and compared against each other. However, this feature is not included in the local version that you have installed. If you are interested in the more advanced features, please contact support@gams.com.

Run the GAMS WebUI on a server

You can also run the GAMS WebUI on a server. This adds multi-user and multi-application support as well as many enterprise features such as authorisation via e.g. LDAP, load balancing, rolling updates and much more. Interested? Please contact us!
