Getting started

Last updated: Jauary 10th, 2019

Introduction

This chapter shows the first steps of how to use the GAMS WebUI and how to deploy your own GAMS models. For using this application a few components have to be installed first, see section Installation.

Start the WebUI

Demo Applications

To get a feeling for the use of the WebUI, it is recommended to have a look at the demo applications first. On windows, the demo applications are located in C:\Users\<user_Name>\GMSWebUI\examples\. If you are using MacOS or Linux, you can find those demo applications here [missing link].

The following demos are available:

  • trnsport
    • This problem looks for a least cost shipping schedule that meets requirements at markets and supplies at factories. More about this transportation Problem, see here.
  • trnsport_live
    • A "live" version of the trnsport model. Enables the specification of locations of suppliers and markets from within the Webui and calculates relevant distances based on geodata.
  • pickstock
    • Optimization model to pick a small subset of the stocks together with some weights, such that this portfolio has a similar behavior to our overall Dow Jones index.
  • pickstock_live
    • A "live" version of the pickstock model. Allows to include current Dow Jones stock prices as well as ones from past periods.
  • kport
    • This problem computes minimal cost solutions satisfying the demand of pre-given product portfolios. More about this optimization Problem, see here.
To start a demo application, you need to run the corresponding GAMS model (e.g. C:\Users\<user_Name>\GMSWebUI\examples\trnsport\trnsport.gms) with the compile-time parameter --webui=launch. This can either be done from within the new GAMS Studio or the GAMSIDE:

or using the command prompt:

cd path/to/gms_file && gams trnsport.gms --webui=launch

Note: To use the command prompt, GAMS must be in the PATH environment variable of your operating system!

Note:

If you run a model with "--webui=launch", all required R packages will be loaded. When you start the application for the first time, the packages are installed first. This may take a while.

Some of the example models included in the GAMS WebUI installation require additional Python packages. These packages should get installed automatically once you run the respective model for the first time. However, in case they do not install properly, try installing them manually: from the command line navigate to your GAMS installation folder and then to the subfolder: GMSPython\Scripts (e.g.: cd C:\GAMS\win64\25.1\GMSPython\Scripts). To install the respective package, enter: pip install <package_name>. Below you find a table with the python packages required for the different example models.

Demo Application Python packages
Pickstock_live pandas, pandas_datareader
Trnsport_live geocoder

WebUI Overview

Let's take the trnsport model as example to get an overview of the functionality of the WebUI. Hierzu starten wir das Modell trnsport.gms mit dem bereits zuvor erwahnten Befehl --webui=launch.

Once all the packages have been loaded, the GAMS WebUI will launch. Below you see the input mask of the WebUI as it appears after the start of the trnsport model.

The screen is essentially divided in two:

Navigation bar:


On the left side you can find the navigation bar. Here you can navigate between the categories:

  1. Input:
    Visualization & configuration of input data for the next calculations.
  2. Output:
    Visualization of calculated output data.
  3. GAMS interaction:
    While solving a model with GAMS this section shows the Log and Lst files.
  4. Compare Scenarios:
    Module for the direct comparison of results / scenarios in a Spit-screeen or a switchable single-scenario view ("tab view").
  5. Advanced Options:
    Download of (temporary) files created during the calculations from the working directory.

In addition, the buttons for loading input data and for starting a GAMS run are located here:

  1. Load data:
    Load input data for a model run. This can be an Excel spreadsheet or existing data from the database.
  2. Solve model:
    Solve the GAMS model with the configured input data. This will start GAMS in the background.
Main window:

The main window always displays the content of the section selected by the user in the navigation bar. Here, for example, data is visualized, scenarios are compared and analyses are evaluated. More about the individual features will be explained now.

Workflow

The typical workflow is quite simple:

  1. Import input data
  2. Adapt the data if desired
  3. Solve GAMS model
  4. Inspect the visualized results
  5. Save the scenario data in the database or discard the results
  • Besides that, already stored results can be re-imported from the database and compared with each other at any time.

Import data

A WebUI usually makes sense if there is data to be visualized, e.g. as tables, diagrams, or other graphics. This applies to output data as well as input data. Input data which should be visualized in the WebUI can be (re-)imported either from existing scenarios in the database or locally via an Excel spreadsheet. To import such data, click on the "Load data" button in the navigation bar. The following window pops up:

You can decide whether to import an existing data set from the database or load an existing spreadsheet from your local machine. When the used GAMS model is started for the very first time in the web application, there is no data in the database yet, so loading a local file is the only reasonable option. For our trnsport model shown here, there is a suitable Excel spreadsheet in the directory from which the model was started, e.g. in C:\Users\<user_Name>\GMSWebUI\examples\trnsport\trnsport.xlsx.

Note:

Every time you start the WebUI for a GAMS model, an Excel spreadsheet is created in the directory of the executed model. This file contains data extracted from the GAMS model. Read more about this topic.

To load the desired Excel spreadsheet, click on the button "load data". Under the menu item "Local" click on "browse". Navigate in the opening Explorer now to the directory of the executed model, e.g. C:\Users\<user_Name>\GMSWebUI\examples\trnsport\trnsport.xlsx, and select the desired file. Now assign a name for the new scenario and confirm the data import with a click on "import". A scenario is now created in the database with the specified name.

Tip:

Instead of the whole spreadsheet you can load single sheets.

Data manipulation

The previously empty tables in the main window are now filled with some data. In the upper part of the main window you can navigate between different tabs to switch between the visualization of declared GAMS symbols. More about the tabular and graphical representation of input data in the WebUI is explained here.
We can now change the input data. If configured accordingly, individual cells can be edited in tables. In addition to tables, input widgets such as sliders, dropdown menus, checkboxes and date selectors are also available. In the trnsport model shown here, the "Input Options" section for manipulating the data is particularly suitable.

Click on the tab "Input Options". With the help of sliders and a dropdown menu we can now adjust the freight costs and the model type to be used. If we are satisfied with the settings we can start the GAMS run.

Solve the model

A click on the "Solve model" button starts the model run. In the background GAMS is called and the model trnsport.gms is started. The values set by us in the previous section now serve as input data for the started GAMS model trnsport.gms.
During the calculations, the WebUI switches to the section "GAMS interaction". There you can see the current GAMS log and lst files.

Note: Since the models of the demo applications are solved very quickly, this step is sometimes hardly visible. The menu entry "GAMS interaction" can also be viewed at any time after a model run.

As long as the calculations are running in GAMS, the "Stop" button on the left can be clicked. This immediately stops the running GAMS process and thus the model calculations.
After the run, the view of the WebUI changes to "Output".

Inspect results

In the "Output" section the results are visualized. As with the input data, a distinction is made between GAMS parameters and scalar values. GAMS parameters are each visualized in a separate tab and scalars are combined in one view.
If a graphical visualization has been configured for a symbol, it is shown first by default. In the trnsport model a graphic is configured for the only present output parameter in the form of a map, see image below.

With a click on the button in the upper right corner you can switch between graphical and tabular data visualization.

All scalar results are summarized in "Output Scalars". The default visualization selected here are tiles. Like before, you can switch between graphical and tabular visualization with a click on the button.

Tip:

All temporarily created files of the model run like the lst and log files can be downloaded separately or as ZIP file in the "Advanced Options" section of the navigation bar.

Save / delete results

When calculating a GAMS model with parameterized input data, we use the term "scenario". A scenario and its results can be stored in the connected database. The interface to the database can be reached in the header of the WebUI by choosing the menu entry "Scenario".

Here, we can choose between the following options:

  • Save
    Save data of an existing scenario in the database.
  • Save As
    Save a scenario in the database by specifying a (new) name.
    Optional tags can be added to the scenario. With the help of assigned tags, scenarios can be identified, independent of the scenario name or the input and output data. Tags represent an additional identification feature of a scenario, on the basis of which the database can be searched at a later time. This is of particular interest in the context of hypoercube jobs.
  • Edit metadata
    Allows you to edit scenario metadata, i.e. the scenario name and specified tags.
    You can also attach files to a scenario here. These can be files of any format. If a file is attached to a scenario, this file is stored as Binary Large Object (blob) in the database and can be accessed again at any time.
  • Export
    Download of the scenario data in the form of an Excel spreadsheet. The spreadsheet contains all tabular input- and output (results) data of the scenario which is displayed in the WebUI.
  • Delete
    Delete a scenario from the database.

Scenario comparison

The scenario comparison module is used to directly compare the input and output data of a small number of scenarios. The data that can be compared here correspond to the data that has already been visualized as input and output data the corresponding GAMS model (see section Data manipulation and Inspect results).
If two scenarios are to be compared, the split-screen display is particularly suitable. In this way, the data can be displayed side by side.

In addition to the split screen mode, several scenarios can also be compared lying side by side in individual tabs (like tabs in browsers). This makes sense if more than 2 scenarios are to be compared. You can switch between split screen mode and the representation of the scenarios in tabs via the switch-button in the navigation bar.

The comparison mode makes comparing scenarios much more efficient and convenient. As soon as this option is activated via the Compare mode button in the navigation bar, switching the displayed data within a scenario is also performed for all other open scenarios. This means that if you change the symbol to be displayed in a scenario (e.g. switch from 'Price' to 'absolute error' in the image above), the view also changes accordingly for the compared scenario(s).

Additional Features

Manage multiple scenarios

The WebUI presented here was designed to configure and solve one scenario at a time. But what if we want to solve a multitude of such scenarios with different parameterizations one after the other in order to investigate the behavior of the solutions more closely? To avoid having to configure and solve each scenario individually, there is the "Hypercube mode". In this interface a model parameter can be set several times. From this multiple parameterization all possible scenarios are generated, which result from the combination of all set parameters. As a consequence, any number of scenarios can be configured and analyzed at once.

We developed this tool to efficiently generate, store and analyze a multitude of scenarios. Look here for more information.

Run the GAMS WebUI on a server

You can also run the GAMS WebUI on a server. This adds multi-user and multi-application support as well as many enterprise features such as authorisation via e.g. LDAP, load balancing, rolling updates and much more. However, this feature is not included in the WebUI version that you have installed. If you are interested in this features, please contact support@gams.com.
